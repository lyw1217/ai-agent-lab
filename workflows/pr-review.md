# PR Review Workflow

## 언제 쓰면 좋은가

- PR의 버그, 회귀, 누락 테스트를 찾을 때
- 리뷰어가 보기 전에 1차 self-review를 할 때
- 큰 diff에서 위험한 변경을 빠르게 찾을 때

## 기본 흐름

1. PR 목적과 변경 범위를 요약한다.
2. diff에서 user-facing behavior 변화를 찾는다.
3. 테스트 누락과 edge case를 확인한다.
4. 보안/성능/호환성 리스크를 본다.
5. actionable comment만 남긴다.

## 리뷰 기준

- 심각한 버그와 회귀를 우선한다.
- 스타일 취향보다 동작 위험을 우선한다.
- 근거 없는 추측성 코멘트를 남기지 않는다.
